# Правила работы с репозиторием

## Ветки

### `prod`
- Содержит только **стабильно работающий код**.
- Используется для **деплоя**.
- В эту ветку можно вливать изменения **только из `dev`**.

### `dev`
- Основная ветка **разработки**.
- В нее вливаются **отдельные ветки разработки**.
- После стабилизации изменений они попадают из `dev` в `prod`.

### Ветки разработки

Создаются для выполнения отдельных задач.

#### Нейминг веток

{отдел}/{смысл}-{название-ветки}


**Параметры:**

- **отдел**
  - `front` — фронтенд
  - `back` — бэкенд
  - если отдел **не указан**, ветка предполагает работу **с обоими отделами**

- **смысл**
  - `feat` — новый функционал
  - `fix` — исправление ошибки
  - `refactor` — рефакторинг

**Примеры:**


- front/feat-user-profile
- back/fix-auth-token
- refactor-api-client


---

## Маршрут изменений

Схема движения изменений между ветками:


feature branch → dev → main


Правила:

- отдельные ветки разработки вливаются в **`dev`**
- в **`main`** вливается **только `dev`**

---

## Нейминг коммитов

Формат:


{отдел}/{смысл}: {название коммита}


**Примеры:**


- front/feat: Добавил страницу профиля
- back/fix: корретная валидация токенов
- refactor: упростил логику апи клиента


---

## Права на merge

Вливать изменения в ветки **`dev`** и **`main`** могут **только тимлиды**:

- Арсений
- Сережа

## Алгоритм запуска проект

1. Установить Docker Desktop.
2. В корне проекта выполнить:

```bash
docker compose up --build
```

Тогда по адресу `http://localhost:8000/docs` появится документация по реализованным ручкам.

login возвращает в ответе выданный access токен и задает его в cookies.
Для проверки ручки me нужно или в документации вставить токен в Authorize, или в формате запроса к серверу добавить заголовок формата 'Authorization: Bearer <access_token>'
